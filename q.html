⚙️ أهم معلمات النواة

تُطبّق معلمات dom0 في GRUB، ومعلمات VMs عبر qvm-prefs

🖥️ معلمات dom0 (تُطبّق في GRUB)

🔴 معلمات Xen: أضفها إلى GRUB_CMDLINE_XEN_DEFAULT

xen_scrub_pages=1

🔴 معلمات Linux: أضفها إلى GRUB_CMDLINE_LINUX

smt=off init_on_free=1 init_on_alloc=1 lockdown=confidentiality module.sig_enforce=1 ipv6.disable=1 noswap slab_nomerge slub_debug=FZ vsyscall=none kernel.kptr_restrict=2 oops=panic debugfs=off quiet loglevel=0
🛡️ smt=off إجراء أمني إلزامي: تعطيل Hyper-Threading يمنع هجمات القنوات الجانبية (Spectre/Meltdown) بين الأجهزة الافتراضية. نظام التحديث في Qubes يفرض هذه المعلمة تلقائياً عند كل تحديث للنواة.
🖼️ معلمات القوالب و AppVMs (تُطبّق عبر qvm-prefs)

هذه المعلمات تحمي برامجك من هجمات المعالج. في Qubes، VMs تستخدم نواة dom0 لكن معاملاتها تُضبط بشكل منفصل ولا ترث معاملات GRUB.

init_on_free=1 init_on_alloc=1 mitigations=auto random.trust_cpu=off random.trust_bootloader=off spec_store_bypass_disable=on tsx_async_abort=full mds=full,nosmt page_alloc.shuffle=1 randomize_kstack_offset=on nosmt=force l1tf=full,force
📖 شرح كل معلمة
# المعلمة الشرح الفائدة يطبق على
1 xen_scrub_pages=1 يمسح الذاكرة قبل إعادة استخدامها بين VMs يمنع تسرب البيانات من VM إلى آخر 🖥️ Xen
2 smt=off يعطل Hyper-Threading يمنع هجمات Spectre/Meltdown بين VMs 🔄 Linux (للجميع)
3 init_on_free=1 يمسح الذاكرة فور تحريرها يمنع تسرب كلمات المرور والمفاتيح 🔄 Linux (للجميع)
4 init_on_alloc=1 يمسح الذاكرة قبل تخصيصها يمنع الوصول إلى بقايا بيانات سابقة 🔄 Linux (للجميع)
5 lockdown=confidentiality يمنع حتى الجذر من تعديل النواة أو قراءة ذاكرتها يمنع العبث بنظام dom0 🖥️ dom0
6 module.sig_enforce=1 يمنع تحميل وحدات نواة غير موقعة يمنع حقن rootkits 🖥️ dom0
7 ipv6.disable=1 يعطل IPv6 بالكامل يمنع تسرب البيانات عبر IPv6 🔄 Linux (للجميع)
8 noswap يعطل الـ swap تماماً يمنع تسرب البيانات إلى القرص 🔄 Linux (للجميع)
9 slab_nomerge يمنع دمج الكائنات المتشابهة في النواة يصعّب استغلال ثغرات الكومة 🔄 Linux (للجميع)
10 slub_debug=FZ يفعل فحوصات سلامة الذاكرة ويضيف مناطق عازلة يكتشف تجاوزات السعة 🔄 Linux (للجميع)
11 vsyscall=none يعطل واجهة vsyscall القديمة يغلق سطح هجوم ROP 🔄 Linux (للجميع)
12 kernel.kptr_restrict=2 يمنع عرض عناوين النواة يصعّب بناء هجمات ROP 🖥️ dom0
13 oops=panic يجعل النواة تنهار عند أي خطأ يمنع استغلال الأخطاء لمواصلة الهجوم 🖥️ dom0
14 debugfs=off يعطل debugfs بالكامل يمنع تسرب معلومات بنية النواة 🖥️ dom0
15 quiet loglevel=0 يمنع عرض رسائل النواة أثناء الإقلاع يمنع تسرب معلومات للمهاجم 🔄 Linux (للجميع)
16 mitigations=auto يفعّل تصحيحات الثغرات تلقائياً يمنع Spectre, Meltdown داخل VMs 📱 AppVMs
17 random.trust_cpu=off لا تثق بمولد الأرقام العشوائية في المعالج يمنع تخمين المفاتيح عبر RNG مخترق 📱 AppVMs
18 random.trust_bootloader=off لا تثق بمولد الأرقام العشوائية في محمل الإقلاع يمنع تخمين المفاتيح عبر RNG مخترق 📱 AppVMs
19 spec_store_bypass_disable=on يعطل ثغرة Spectre v4 يمنع تسرب البيانات عبر المخازن المؤقتة 📱 AppVMs
20 l1tf=full,force يفعل الحماية الكاملة من L1 Terminal Fault يمنع تسرب بيانات ذاكرة L1 📱 AppVMs

✅ الحل الموصى به: راوتر WISP + Ethernet فقط

🛡️ راوتر WISP + Ethernet فقط (الإخفاء التام لجهاز Qubes)

راوتر صغير (مثل GL.iNet) في وضع WISP، متصلاً بجهاز Qubes عبر كابل Ethernet فقط، مع تعطيل البث اللاسلكي للشبكة الداخلية.

💡 الفكرة الأساسية:
  • الراوتر يلتقط الإنترنت من شبكة الموزع الوسيط لاسلكياً
  • لكنه لا يبث أي إشارة WiFi داخلية في منزلك
  • جهاز Qubes متصل به عبر كابل Ethernet فقط
  • النتيجة: جهاز Qubes غير موجود على أي شبكة لاسلكية، والآخرون لا يرون إلا راوتراً عادياً

🔧 خطوات الإعداد

  1. اشتر راوتر صغير يدعم وضع WISP (مثل GL.iNet أو TP-Link)
  2. كوّن الراوتر على وضع WISP، واتصل بشبكة الموزع، مع تفعيل MAC Randomization
  3. عطّل البث اللاسلكي الداخلي تماماً
  4. وصل الراوتر بجهاز Qubes عبر كابل Ethernet — سيظهر في Qubes كـ eth0
✅ النتيجة النهائية: الموزع والجيران يرون راوتراً واحداً فقط. جهاز Qubes غير مرئي لأي طرف على الشبكة المحلية.

🧱 بنية الشبكة داخل Qubes: sys-net و sys-firewall

🧱 لماذا يوجد جداران؟

في Qubes، لا أحد يتصل بالإنترنت مباشرة إلا sys-net. ثم يأتي sys-firewall كوسيط إجباري لجميع الأجهزة الأخرى.

│ eth0 (واجهة فيزيائية)
🧱 sys-net
غير موثوق — متصل بالعالم الخارجي
│ vif (واجهة افتراضية)
🧱 sys-firewall
موثوق — جدار ناري داخلي للجميع
├──────────────┬──────────────┐
🧅 sys-whonix 📦 أي Qube
⚠️ خطأ شائع قاتل: لا تقم أبداً بتوصيل أي Qube مباشرة بـ sys-net. الإعداد الصحيح: التطبيق ← sys-firewall ← sys-net ← الإنترنت.

⚠️ سيناريو: ماذا لو تم اختراق sys-net؟

🎭 أسوأ الاحتمالات: مخترق يتحكم في الراوتر
✅ ما يستطيع فعله:
  • رؤية حركة المرور (لكنها مشفرة عبر Tor)
  • قطع الاتصال أو حجب مواقع
  • محاولة مهاجمة sys-firewall (يحتاج ثغرة Xen نادرة)
❌ ما لا يستطيع فعله:
  • الوصول إلى dom0 أو واجهة المستخدم
  • رؤية شاشتك أو تسجيل لوحة المفاتيح
  • الوصول إلى باقي الأجهزة الافتراضية
  • معرفة أنك تستخدم Qubes (في إعداد WISP)
  • فك تشفير حركة Tor
🛡️ الخلاصة: حتى في أسوأ سيناريو، يبقى الضرر محدوداً داخل sys-net. باقي النظام يبقى آمناً بفضل عزل Qubes.

📚 خلاصة تبويب الشبكة:
  • الحل الموصى به: راوتر WISP + Ethernet فقط + تعطيل البث اللاسلكي الداخلي.
  • بنية Qubes: sys-net (غير موثوق) ← sys-firewall (موثوق) ← باقي الأجهزة. لا تتجاوز sys-firewall أبداً.
  • عدوك الحقيقي: الموزع الوسيط والجيران على الشبكة المحلية — وليس المزود الرسمي.
  • مسار البيانات: Qubes ← راوتر WISP ← موزع ← مزود إنترنت ← موقع نهائي. البيانات تمر مرة واحدة في كل اتجاه.
  • سيناريو الاختراق: حتى لو تم اختراق sys-net، لا يستطيع المهاجم الوصول لـ dom0 أو باقي الأجهزة إلا بثغرة Xen نادرة.
  • خطط الطوارئ: Ethernet (🟢) → USB Tethering (🟡) → WiFi داخلي مشفر (🟡) → WiFi مباشر مع حماية (🟠) → WiFi عادي (🔴).
  • طبقة VPN إضافية: يمكن تشغيل VPN على راوتر WISP نفسه لحماية البيانات حتى قبل وصولها لـ Qubes.
  • الكروت الملصقة: توفر خصوصية من الموزع والجيران، لكن في تحقيق جنائي رسمي قد تُتَتبع عبر كاميرات المحل.

  • الخطوة 1 (داخل Gateway): Gateway Tor يقوم بتشفير البيانات (المشفرة فعلاً بـ TLS) ثلاث مرات متتالية. يبدأ بمفتاح عقدة الخروج، ثم الوسطى، ثم الدخول. هذا يغلفها بأربع طبقات إجمالاً.
  • الخطوة 2 (عند عقدة الدخول): تفك تشفير طبقة Tor الخارجية، وتكشف عن رسالة مشفرة بـ 3 طبقات (1 TLS + 2 Tor).
  • الخطوة 3 (عند العقدة الوسطى): تفك تشفير طبقة Tor الوسطى، وتكشف عن رسالة مشفرة بـ 2 طبقات (1 TLS + 1 Tor).
  • الخطوة 4 (عند عقدة الخروج): تفك تشفير طبقة Tor الأخيرة، لتكشف عن رسالة مشفرة بطبقة TLS/HTTPS الواحدة التي لا تستطيع فكها، فترسلها للخادم النهائي الذي يمتلك مفتاح TLS.

📖 ملاحظة فنية متقدمة: بهذا الشكل، حتى لو كانت عقدة الخروج خبيثة، فهي لا ترى سوى بيانات مشفرة بتقنية TLS، ولا تستطيع التجسس على المحتوى الفعلي لاتصالك (كلمات المرور، الرسائل، إلخ). يستبدل Tor حالياً خوارزمية "tor1" القديمة بخوارزمية "Counter Galois Onion (CGO)" الأحدث والأكثر مقاومة للهجمات.

🛡️ لماذا Whonix Workstation؟ ليس مجرد توجيه مرور

من وثائق Whonix و Privacy Guides: Whonix-Workstation ليست مجرد آلة افتراضية عادية، بل هو نظام تشغيل مُصلب (Hardened OS) يعتمد على توزيعة Kicksecure الأمنية. هذا يوفر طبقات حماية إضافية تتجاوز بكثير مجرد تمرير البيانات عبر Tor.

إذا تم اختراق المتصفح أو أي تطبيق، فإن المهاجم لن يتمكن من معرفة عنوان IP الحقيقي، ولكن يمكنه الوصول إلى كل البيانات داخل الجهاز. هنا يأتي دور التصليب الداخلي للمساعدة في منع أو إعاقة هذا الاختراق من البداية.

  • عزل التدفق (Stream Isolation): كل تطبيق يحصل على دائرة Tor خاصة به، مما يمنع ربط هوياتك المختلفة.
  • مكافحة بصمة لوحة المفاتيح (Keystroke Anonymization): بفضل أداة kloak، يتم تشويش أسلوب كتابتك لمنع التعرف عليك من خلال إيقاع الكتابة.
  • تشفير الذاكرة المؤقتة (Encrypted Swap): حتى البيانات التي تخرج مؤقتاً إلى القرص تبقى مشفرة بالكامل.
  • مُخصص ذاكرة مُصلب (Hardened Memory Allocator): يحمي من استغلال ثغرات الذاكرة المعقدة.
  • حماية النواة (Kernel Hardening): يتم تطبيق إعدادات sysctl المتقدمة بشكل افتراضي. ويمكن تعزيزها لاحقاً عبر تثبيت hardened-kernel الذي يعيد بناء النواة برموز عشوائية لتصعيب استغلال الثغرات.
نتيجة هذه التحصينات: Whonix-Workstation يُعتبر بيئة عالية الأمان بحد ذاته. حتى لو تم اختراقه، فإن تحصيناته الداخلية تجعل مهمة استمرار واستغلال هذا الاختراق أصعب بكثير. يوصى دائمًا باستنساخه بشكل دوري والعمل على نسخ نظيفة لتقليل أثر أي اختراق محتمل.

🛡️ المشكلة: حراس الدخول (Entry Guards) ثابتون

حارس الدخول هو أول عقدة Tor تتصل بها بوابة Whonix. وفقاً لوثائق Whonix:

"من أجل تحسين إخفاء الهوية، قرر مطورو Tor تقليل عدد حراس الدخول من 3 إلى 1 وجعل دورة تغيير الحراس أقل تكراراً (4-8 أسابيع، قد تصل إلى 120 يوماً)."
⚠️ النتيجة: إذا وقع اختيارك على عقدة دخول خبيثة، فستظل عالقاً معها لأشهر. كما أن استخدام Gateway واحد لجميع الأنشطة يعني أن جميع هوياتك مرتبطة بنفس حارس الدخول.

🌐 شرح أنواع اتصالات Tor و VPN

قبل المقارنة، دعنا نشرح معنى كل نوع من أنواع الاتصال:

🧅 Tor فقط (الأساسي)

ماذا يعني؟ جهازك يتصل مباشرة بشبكة Tor دون أي طبقات إضافية.

جهازك ← شبكة Tor ← الإنترنت

ماذا يرى ISP؟ يرى أنك تتصل بشبكة Tor (يمكنه معرفة أنك تستخدم Tor).

ماذا يرى الموقع؟ يرى عنوان IP لعقدة خروج Tor.

في Qubes: anon-work → sys-whonix → sys-firewall → sys-net

🌉 Tor + Bridges (لتجاوز الحجب)

ماذا يعني؟ مثل Tor فقط، لكنك تتصل أولاً بـ "جسر" (Bridge) يخفي حقيقة أنك تستخدم Tor.

جهازك ← جسر سري ← شبكة Tor ← الإنترنت

ماذا يرى ISP؟ يرى اتصالاً عادياً مشفراً (لا يعرف أنه Tor).

متى تستخدمه؟ إذا كان Tor محجوباً في بلدك (مثل الصين، إيران، روسيا).

أنواع الجسور: obfs4, Snowflake, meek_lite

🔐 VPN → Tor (الأكثر أماناً)

ماذا يعني؟ تتصل أولاً بـ VPN، ثم من VPN إلى شبكة Tor.

جهازك ← VPN ← شبكة Tor ← الإنترنت

ماذا يرى ISP؟ يرى اتصال VPN فقط (لا يعلم أنك تستخدم Tor).

ماذا يرى VPN؟ يرى أنك تتصل بشبكة Tor (لكنه لا يعرف وجهتك النهائية).

ماذا يرى الموقع؟ يرى عنوان Tor فقط.

في Qubes: anon-work → sys-whonix → sys-vpn → sys-firewall → sys-net

🔄 Tor → VPN (للراحة فقط)

ماذا يعني؟ تتصل أولاً بـ Tor، ثم تخرج إلى الإنترنت عبر VPN.

جهازك ← شبكة Tor ← VPN ← الإنترنت

ماذا يرى ISP؟ يرى أنك تتصل بشبكة Tor.

ماذا يرى VPN؟ لا يعرف هويتك الحقيقية (لأنك أتيت من Tor)، لكنه يرى وجهتك النهائية.

ماذا يرى الموقع؟ يرى عنوان VPN (وليس Tor).

متى تستخدمه؟ إذا كنت تريد الوصول لخدمات تمنع Tor (مثل بعض البنوك)، أو تريد IP ثابت.

⚠️ تحذير: VPN يرى وجهتك النهائية! استخدم هذه الطريقة فقط إذا كنت تثق بمزود VPN ثقة كاملة.
🔒 VPN فقط (للاستخدام العادي)

ماذا يعني؟ تتصل مباشرة بـ VPN بدون Tor.

جهازك ← VPN ← الإنترنت

ماذا يرى ISP؟ يرى اتصال VPN فقط.

ماذا يرى VPN؟ يرى كل شيء: هويتك (عنوان IP الحقيقي) ووجهتك النهائية.

ماذا يرى الموقع؟ يرى عنوان VPN.

⚠️ تحذير: مزود VPN يعرف هويتك الحقيقية وكل المواقع التي تزورها! مناسب فقط للاستخدام العادي غير الحساس.
❌ بدون Tor/VPN (غير آمن)

ماذا يعني؟ اتصال مباشر بالإنترنت دون أي حماية.

جهازك ← الإنترنت

ماذا يرى ISP؟ يرى كل شيء: كل موقع تزوره، كل ما تفعله.

ماذا يرى الموقع؟ يرى عنوان IP الحقيقي الخاص بك.

🔴 لا تستخدم هذا أبداً للأنشطة الحساسة!

المسار: جهازك ← VPN ← شبكة Tor ← الإنترنت

ISP يرى: VPN | VPN يرى: Tor | الموقع يرى: IP Tor

التوصية: الأكثر خصوصية - ISP لا يعرف أنك تستخدم Tor.

🔄 Tor → VPN

المسار: جهازك ← شبكة Tor ← VPN ← الإنترنت

ISP يرى: Tor | VPN يرى: وجهتك | الموقع يرى: IP VPN

⚠️ تحذير: VPN يرى وجهتك النهائية! للراحة فقط مع خدمات تمنع Tor.

💡 ملاحظات هامة:
  • 🔹 VPN → Tor: الأكثر خصوصية - ISP لا يعرف أنك تستخدم Tor، وVPN لا يعرف وجهتك النهائية.
  • 🔹 Tor → VPN: للراحة فقط - يسمح بالوصول لخدمات تمنع Tor، لكن VPN يرى وجهتك (تحتاج لثقة كاملة بالمزود).
  • 🔹 Tor + Bridges: ضروري في الدول التي تحجب Tor. استخدم Snowflake أو obfs4.
  • 🔹 VPN فقط: مناسب للاستخدام العادي غير الحساس، لكن مزود VPN يعرف هويتك وكل وجهاتك.
  • 🔹 في Qubes: لتطبيق VPN → Tor، اجعل sys-whonix يستخدم sys-vpn كـ NetVM.

🌉 ما هي الجسور (Bridges)؟ - من وثائق Whonix

الجسور (Tor Bridge Relays) هي نقاط دخول بديلة إلى شبكة Tor، تعمل كبديل لحراس الدخول العاديين. استخدام الجسر يجعل من الصعب على مزود الخدمة (ISP) معرفة أنك تتصل بـ Tor.

🕵️ ماذا يرى المراقب؟

  • بدون جسر: يرى اتصالاً بشبكة Tor ← قد يحجبه
  • مع جسر: يرى اتصالاً مشفراً عادياً ← لا يمكنه تمييزه بسهولة
📖 ملاحظة: الجسور ليست رصاصة فضية. إنها تجعل الاكتشاف أصعب، لكنه ليس مستحيلاً.

👥 من يحتاج إلى استخدام الجسور؟

معظم مستخدمي Whonix لا يحتاجون جسوراً. لكن ثلاثة فئات قد تحتاجها:

الفئةالهدفنوع الجسر المناسب
1️⃣ المحجوبونTor محجوب - يحتاج أي طريقة للوصولobfs4 أو Snowflake
2️⃣ المخفيونإخفاء حقيقة استخدام Tor (الخصم خطير)جسور خاصة (Private)
3️⃣ المختبروناختبار اتصال الجسورأي نوع
⚠️ تنبيه: إذا كنت في دولة تمنع Tor (مثل الصين، إيران، روسيا)، فأنت بحاجة إلى جسر. استخدم Snowflake أو meek_azure.

🎛️ Tor Control Panel - أسهل طريقة لتفعيل الجسور

الطريقة التي يعرفها الجميع:

  1. في sys-whonix، انقر على أيقونة Tor Control Panel في شريط المهام (البصلة 🧅).
  2. انقر على Configure.
  3. اذهب إلى تبويب Tor Bridges.
  4. اختر نوع الجسر:
    • None - بدون جسر (افتراضي)
    • obfs4 - الجسر الموصى به
    • Snowflake - للمناطق شديدة الحجب
    • meek_lite - للصين والدول القمعية
    • Custom - لإدخال جسر يدوياً
  5. انقر Connect وانتظر حتى يظهر "Connected to Tor".

📸 كيف يبدو Tor Control Panel؟

  • أيقونة البصلة 🧅 في شريط المهام السفلي (LXQt)
  • تظهر حالة الاتصال: "Connected to Tor" أو "Connecting..."
  • زر Configure يفتح نافذة الإعدادات
🖥️ أو عبر الطرفية: anon-connection-wizard
💡 للحصول على جسور مخصصة:
  • موقع bridges.torproject.org
  • أو أرسل بريداً إلى bridges@torproject.org من Gmail أو Riseup
  • انسخ الجسر والصقه في خيار Custom
❄️ Snowflake - الجسر الموصى به للمناطق المحجوبة

من وثائق Whonix: Snowflake هو نقل قابل للتوصيل (Pluggable Transport) يمرر حركة المرور عبر وكلاء مؤقتين باستخدام WebRTC.

✅ مميزات Snowflake:

  • يعمل حتى في البيئات شديدة الحجب
  • يستخدم متطوعين حول العالم كوكلاء
  • لا يحتاج إلى خادم مركزي
  • مقاوم للـ Active Probing

🔧 للإعداد اليدوي (متقدم): أضف إلى /usr/local/etc/torrc.d/50_user.conf:

UseBridges 1
ClientTransportPlugin snowflake exec /usr/bin/snowflake-client

✅ التحقق من أنك تستخدم جسراً (Nyx)

للتأكد من أن اتصالك يمر عبر جسر وليس عبر Guard عادي:

  1. في sys-whonix، افتح Nyx من قائمة التطبيقات أو اكتب nyx في الطرفية.
  2. استخدم السهم الأيمن للانتقال إلى الصفحة 2 من 5.
  3. انظر إلى معلومات الدائرة (Circuit):

✅ تستخدم جسراً:

192.168.0.1 UNKNOWN 1 / Guard

❌ لا تستخدم جسراً:

123.45.67.89 RelayName 1 / Guard

للخروج من Nyx: اضغط q مرتين.

⚠️ تحذير: Bridged Networking ≠ Bridges

من وثائق Whonix: هناك شيئان مختلفان تماماً رغم تشابه الاسم:

المصطلحالمعنى
Bridges (الجسور)نقاط دخول بديلة لشبكة Tor لتجاوز الحجب
Bridged Networkingإعداد في برامج المحاكاة (VirtualBox) لا يجب تغييره
🚫 لا تغير إعدادات الشبكة في VirtualBox لـ Whonix-Gateway إلى Bridged Network! هذا غير مدعوم وغير مختبر وقد يكسر أمانك.
كيفية الإعداد في Whonix:
1. افتح Tor Controller Panel من الشريط العلوي داخل sys-whonix
2. اختر Configure
3. اختر Bridge Type
4. اختر نوع الجسر المناسب (يُفضل Snowflake للمبتدئين)
5. يمكنك الحصول على جسور إضافية عبر: bridges@torproject.org أو موقع Tor Project
نصيحة متقدمة: للبيئات شديدة الحجب، يمكنك استخدام Snowflake لأنه الأصعب في الاكتشاف والحجب، ويعتمد على شبكة واسعة من المتطوعين حول العالم.
📚 خلاصة سريعة:
  • ✅ Gateway = --class AppVM --template whonix-gateway-18 + provides_network true (🔒 بداية التشفير)
  • ✅ Workstation = --class AppVM --template whonix-workstation-18 + netvm sys-whonix-X
  • ✅ جدار ناري مخصص = لكل Gateway جدار ناري خاص به (sys-firewall-A و sys-firewall-B) - يمرر البيانات كما هي
  • ✅ فك التشفير = يحدث فقط في عقد Tor الثلاث (Entry Guard ← Middle Relay ← Exit Node)
  • ✅ Whonix يخفي التوقيت = SDWDate يمنع هجمات التوقيت عملياً
  • ✅ الجسور (Bridges) = تخفي استخدام Tor عن ISP
  • ✅ بوابات متعددة = أنشئ Gateways جديدة من القالب (وليس Clone)
  • ✅ كل Gateway له عقد دخول مختلفة (private volume مستقل)
  • 🔴 لا تشغل بوابتين معاً (للأمان النظري، مع الجسور الخطر ضئيل جداً)
⚠️ تحذير بشأن SSD وTRIM: لا تستخدم الأحجام المخفية (Hidden Volumes) مع SSD إذا كنت تحتاج إنكاراً معقولاً حقيقياً، وذلك بسبب تقنية TRIM وخاصية تسوية التآكل (Wear-Leveling) التي تترك آثاراً محتملة.
📌 الحل الأمثل على SSD: استخدم طريقة التشفير المزدوج (VC خارجي + LUKS داخلي) مع فصل التخزين عن التنفيذ كما هو موضح أدناه.
📦 صندوق الأمانات - التخزين المزدوج (VeraCrypt + LUKS)

الفكرة من Hitchhiker's Guide: طبقة دفاع ضد الإكراه القانوني (Plausible Deniability).

ماذا يرى الخصم إذا أرغمك؟

تدخل كلمة مرور "الفخ" (Decoy Password) → يظهر له ملفات عادية (أفلام، صور شخصية، مستندات عمل عادية).

✅ لا يعرف بوجود البيانات الحساسة أصلاً.

ماذا يوجد حقاً؟

تدخل كلمة المرور الحقيقية → يظهر LUKS داخلي يحوي بياناتك الحساسة (مفاتيح، محادثات، مستندات سرية).

⚠️ لا تستخدم هذه الطريقة على SSD إذا كان الإنكار المعقول ضرورياً (بسبب TRIM).

🔧 كيفية الإنشاء (نظرة عامة):

  1. أنشئ حاوية VeraCrypt عادية بحجم كبير (مثلاً 20GB).
  2. اختر إنشاء "Hidden Volume" داخل الحاوية (مثلاً 5GB).
  3. في الحجم المخفي، أنشئ حاوية LUKS (ملف عادي) أو قسم.
  4. استخدم LUKS للتخزين اليومي الحساس.
تذكر: الفصل بين التخزين والتنفيذ (RAM Pool) يبقى هو الطبقة الأهم.

• الإنكار المعقول: VC يخفي وجود المحادثات أصلاً، بينما LUKS لا يوفر هذه الميزة.
• التوصية للمراسلة شديدة الحساسية: استخدم VC خارجي + LUKS داخلي – يجمع بين إخفاء وجود المحادثات (VC) ومنع التلاعب بالرسائل (LUKS).

🔐 الطريقة الآمنة: فصل التخزين عن التنفيذ (لمنع التسريب)

الفكرة الأساسية: بعد فتح التشفير (VC + LUKS) داخل كيوبس التخزين، لا تقم بتثبيت (mount) نظام الملفات فيه. بدلاً من ذلك، مرر الجهاز المفتوح فقط إلى كيوبس آخر يعمل على RAM Pool. بهذه الطريقة، البيانات الحساسة تُكتب فقط في الذاكرة العشوائية (RAM) وتختفي تماماً عند الإغلاق.

📌 لماذا هذا مهم؟
• إذا قمت بتثبيت نظام الملفات في كيوبس التخزين، فإن البيانات تُكتب على القرص (Swap, logs, cache) وقد تُستعاد لاحقاً.
• إذا قمت بتمرير الجهاز فقط (بدون تثبيت) إلى كيوبس على RAM Pool، فإن البيانات تعمل في RAM فقط وتختفي عند الإغلاق.
• هذه هي الطريقة الوحيدة لمنع التسريب بالكامل في Qubes OS.

  1. في storage-vault: افتح VeraCrypt ثم LUKS (بدون mount). الجهاز الآن جاهز للتمرير.
  2. في dom0: افتح قائمة "Qubes Devices" (أيقونة USB في شريط المهام أو من Qubes Tools).
  3. ستجد الجهاز (مثل secret-data) مدرجاً تحت storage-vault.
  4. حدد الجهاز، ثم اختر كيوبس الوجهة (مثل ram-work) واضغط Attach أو توصيل.
  5. في ram-work: قم بتثبيت الجهاز (mount) كالمعتاد عبر المدير أو الأمر sudo mount /dev/xvdi /mnt/secret.
💡 لماذا هذا سهل؟
• dom0 يتعامل مع أي جهاز بلوك (block device) بنفس الطريقة، سواء كان USB فعلياً أو جهاز LUKS مفتوحاً.
• لا تحتاج إلى كتابة أوامر معقدة – كل شيء يتم عبر الواجهة الرسومية.
• كيوبس الوجهة يرى الجهاز كقرص عادي (مثل /dev/xvdi) ويمكنه تثبيته فوراً.
⚠️ تذكر: لا تقم بتثبيت (mount) الجهاز في كيوبس التخزين. فقط افتح التشفير واتركه. التثبيت يتم فقط في كيوبس الوجهة الذي يعمل على RAM.

📌 الطريقة الأساسية: LUKS فقط (الموصى به للاستخدام اليومي)

مثال لتهيئة LUKS2 مع التكامل (AES-256-XTS + HMAC-SHA256):

sudo cryptsetup luksFormat --type luks2 \ --cipher aes-xts-plain64 \ --integrity hmac-sha256 \ --key-size 512 \ --pbkdf argon2id \ --pbkdf-memory 4194304 \ --iter-time 8000 \ --pbkdf-parallel 4 \ /home/user/data.img
شرح المعاملات:
  • --type luks2 : استخدام النسخة الحديثة من LUKS.
  • --cipher aes-xts-plain64 : وضع التشفير XTS-AES (مفتاح 512 بت).
  • --integrity hmac-sha256 : تفعيل التحقق من السلامة (يمنع التلاعب).
  • --pbkdf argon2id : الأقوى ضد هجمات GPU/ASIC.
  • --pbkdf-memory 4194304 : 4 جيجابايت ذاكرة لاشتقاق المفتاح.
  • --iter-time 8000 : 8 ثوانٍ لفتح الحاوية (يمكن رفعه).

📖 شرح المصطلحات:
  • Wear Leveling: تقنية في SSD توزع عمليات الكتابة على جميع الخلايا بالتساوي لإطالة عمر القرص. هذا يعني أن الكتابة فوق ملف لا تحدث في نفس المكان الفيزيائي.
  • TRIM: أمر يصدره نظام التشغيل ليخبر SSD أن صفحات معينة لم تعد مستخدمة ويمكن مسحها.
  • Garbage Collection: عملية داخلية في SSD تقوم بمسح الصفحات المهملة فعلياً في الخلفية.
  • ATA Secure Erase: أمر يصدر للمتحكم لمسح جدول التعيينات (mapping table) مما يجعل البيانات غير قابلة للوصول.
  • Crypto Erase: تغيير مفتاح التشفير الداخلي للقرص (لأقراص SED) مما يجعل كل البيانات غير قابلة للقراءة.
⚠️ تحذير هام:
  • • لا تستخدم أدوات المسح التقليدية (مثل shred أو wipe) على SSD - فهي غير فعالة وقد تقلل من عمر القرص.
  • • إذا كنت تستخدم تشفير كامل للقرص (LUKS/Veracrypt)، فإن Crypto Erase (تغيير كلمة المرور وإعادة التشفير) هو أسهل طريقة لمسح SSD.
  • • تأكد من تفعيل TRIM على نظامك: sudo systemctl enable fstrim.timer (Linux) أو استخدم Optimize في Windows.
🧹 أدوات إزالة البيانات الوصفية (Metadata Removal)

البيانات الوصفية (Metadata) هي معلومات مخفية في الملفات تكشف تفاصيل عنك: الموقع، الكاميرا، تاريخ الإنشاء، اسم المؤلف، وغيرها. هذه الأدوات تساعدك في إزالتها قبل مشاركة الملفات.

الأداة المنصة نوع الملفات الاستخدام التوصية
ExifTool Windows / Linux / macOS صور، PDF، Office، فيديو، صوت exiftool -all= file.jpg ⭐ الأقوى
ExifCleaner Windows / Linux / macOS صور، PDF، مستندات واجهة رسومية - سحب وإفلات ⭐ سهل الاستخدام
mat2 Linux (Whonix/Tails) صور، PDF، Office، صوت، فيديو mat2 file.jpg ⭐ مدمج في Whonix/Tails
Dangerzone Windows / Linux / macOS PDF، Office، صور يحول الملف إلى PDF آمن (صورة متسلسلة) 🌐 للملفات المشبوهة
PDF Redact Tools Linux PDF فقط سطر أوامر لتحويل PDF إلى صور متخصص للـ PDF

أمثلة سريعة لـ ExifTool

عرض metadata:

exiftool image.jpg

حذف كل metadata:

exiftool -all= image.jpg

حذف من مجلد كامل:

exiftool -all= -r /path/to/folder/

تثبيت سريع

Linux (Debian/Ubuntu):

sudo apt install exiftool mat2

Windows: حمل من exiftool.org أو exifcleaner.com

macOS:

brew install exiftool
💡 نصائح مهمة:
  • • ExifTool هو الأداة الأقوى والأكثر شمولاً - مثبت مسبقاً في Whonix و Tails.
  • • ExifCleaner خيار ممتاز للمبتدئين بواجهة رسومية بسيطة.
  • • mat2 موصى به من PrivacyGuides ومدمج في Whonix Workstation.
  • • Dangerzone يحول أي ملف إلى PDF آمن (يحوله لصور ثم يعيد تجميعه) - مثالي للملفات المشبوهة.
  • • تأكد دائماً من إزالة metadata قبل مشاركة أي ملف، خاصة الصور والمستندات.
⚠️ تذكر: إزالة metadata لا تضمن إزالة العلامات المائية (watermarks) أو تقنيات إخفاء المعلومات (steganography). للملفات شديدة الحساسية، استخدم Dangerzone لتحويلها إلى صور ثم PDF.
🛠️ نصائح إضافية للتخزين الآمن
  • 🔹 فصل الرأس (Header): يمكنك حفظ رأس LUKS على جهاز منفصل – بدون الرأس لا يمكن فتح الحاوية حتى مع كلمة المرور.
  • 🔹 PIM مرتفع في VC: ارفع PIM إلى 500+ لزيادة مقاومة PBKDF2 لهجمات GPU.
  • 🔹 تحذير SSD: لا تستخدم الأحجام المخفية على SSD إذا كان الإنكار المعقول ضرورياً.
  • 🔹 النسخ الاحتياطي للرأس: sudo cryptsetup luksHeaderBackup /dev/sdX --header-backup-file /path/to/backup
  • 🔹 تعطيل TRIM: veracrypt --mount-options=nokernelcrypto /path/to/container /mnt/point
  • 🔹 التمرير دون تثبيت: تذكر دائماً: لا تثبت (mount) في كيوبس التخزين، فقط مرر الجهاز عبر واجهة dom0.
  • 🔹 تنظيف RAM Pool: بعد الانتهاء، يمكنك إيقاف كيوبس ram-work وتفريغ الذاكرة: sudo umount /home/user/ram_pool
نصيحة: استخدم أمر sudo dmesg | tail بعد إدخال USB لرؤية اسم الجهاز (مثل /dev/sdb1) إذا كنت تريد التعامل مع أجهزة USB مباشرة.
✅ ملخص التدفق الآمن:
storage-vault (يفتح VC+LUKS، لا يثبت) → يظهر الجهاز في dom0 → توصيله عبر واجهة dom0 إلى ram-work (يعمل على RAM) → تثبيت الجهاز في ram-work → العمل على البيانات في RAM فقط → عند الإغلاق، تختفي كل البيانات.

🛡️ القاعدة الذهبية: تمرير البلوك ديفايس (جهاز التخزين الخام)

عندما تريد إتاحة جهاز تخزين لأحد الكيوبسات في Qubes، استخدم تمرير البلوك ديفايس (Block Device) بدلاً من تمرير منفذ USB بالكامل. هذه هي الطريقة الموصى بها والمفضلة من قبل فريق Qubes لأسباب أمنية.

💡 فهم الفرق: sda (القرص كاملاً) vs sda3 (قسم واحد)

🔴 sda (القرص كاملاً) – خطر

الـ VM ترى القرص بالكامل وتستطيع:

  • قراءة الأقسام الأخرى
  • تنفيذ أوامر ATA/NVMe خطيرة (مثل Secure Erase)
  • تدمير جدول الأقسام
  • معرفة أنواع الأقسام الأخرى (swap، LUKS...)
🟢 sda3 (قسم واحد) – آمن

الـ VM ترى فقط بيانات القسم:

  • لا تعرف بوجود أقسام أخرى
  • لا تستطيع الوصول للأوامر منخفضة المستوى
  • معزولة تماماً عن باقي القرص
  • تتعامل مع القسم كـ "ملف كبير" فقط

مرر أقل قدر ممكن – القسم الواحد آمن، القرص الكامل خطر

📖 من وثائق Qubes:
"إن تمرير جهاز USB بالكامل يوفر سطح هجوم أكبر مقارنة بتمرير جهاز بلوك ديفايس واحد. وبالمثل، فإن تمرير جهاز بلوك ديفايس بالكامل (القرص بأكمله) يوفر سطح هجوم أكبر من تمرير قسم واحد فقط."
🖥️ مثال عملي:
qvm-block attach work-vm dom0:/dev/sdb1

هذا يمرر القسم الأول فقط من القرص الخارجي إلى VM باسم work-vm.

⚠️ لا تستخدم: qvm-block attach work-vm dom0:/dev/sdb (هذا يمرر القرص كاملاً).

⚠️ استثناء: الأجهزة المركبة (Compound Devices)
بعض أجهزة USB (مثل كاميرات الويب) تظهر كجهاز مركب يحتوي على عدة وظائف (فيديو + ميكروفون). تمرير البلوك ديفايس لا يعمل معها. في هذه الحالة فقط، استخدم sys-usb مخصص مع qvm-usb attach.
💡 القاعدة الذهبية:
• دائماً، مرر أقل قدر ممكن من الجهاز للحصول على أعلى درجة أمان.
• للتخزين: استخدم qvm-block attach vm-name dom0:/dev/sdX# (قسم واحد).
• للأجهزة الخاصة (كاميرات، YubiKey): قد تحتاج qvm-usb attach، لكن كن واعياً لسطح الهجوم الأكبر.

📊 الرسم البياني: التدفق الآمن للبيانات

يوضح الرسم البياني أدناه الفرق بين الطريقة التقليدية (تسريب البيانات) والطريقة الآمنة (فصل التخزين عن التنفيذ) التي يستخدمها هذا الدليل.

❌ الطريقة الخطرة (تسريب البيانات)
📦 حاوية VC → 🔓 فتح VC+LUKS → 💾 Mount + كتابة → ⚠️ بيانات على HDD/SSD
البيانات تُكتب على القرص → يمكن استرجاعها لاحقاً
الطريقة الآمنة (هذا الدليل)
📦 storage-vault → 🔓 فتح VC+LUKS (بدون mount) → 🖱️ تمرير الجهاز عبر dom0 → 🧠 ram-work (RAM Pool) → ✅ Mount + كتابة في RAM
البيانات تُكتب في الذاكرة فقط → تختفي بعد الإغلاق 🧹
🔑 المفتاح هنا
🔄 dom0 يمرر المرجع فقط
🧹 لا أثر على القرص
📌 شرح الرسم البياني:
• الطريقة الخطرة: فتح التشفير وتركيبه (mount) في نفس الكيوبس يؤدي إلى كتابة البيانات على القرص الصلب، مما يترك آثاراً يمكن استرجاعها.
• الطريقة الآمنة: فتح التشفير في كيوبس معزول (storage-vault) ثم تمرير الجهاز المفتوح فقط إلى كيوبس يعمل على RAM Pool (ram-work). البيانات تُكتب في الذاكرة وتختفي عند الإغلاق.
💾 سيناريو عملي: فلاشة LUKS ← Whonix (GNOME Disks)

التدفق الكامل بخطوات بسيطة:

فلاشة LUKS → storage-vault → GNOME Disks (فتح) → تمرير GUI → Whonix → GNOME Disks (Mount)
1 أوصل الفلاشة إلى storage-vault من أيقونة USB
2 في storage-vault: افتح GNOME Disks ← Unlock القسم ← لا تضغط Mount
3 من أيقونة USB: اختر الجهاز المفتوح ← Attach to whonix-workstation
4 في whonix-workstation: افتح GNOME Disks ← Mount
🛡️ لماذا آمن؟ كلمة مرور LUKS لا تغادر storage-vault. Whonix ترى فقط البيانات بعد فك التشفير.
🔄 بعد الانتهاء: Unmount في Whonix ← افصل الجهاز ← Lock في storage-vault ← افصل الفلاشة.

🎯 كيف يعمل؟

  1. الخصم يراقب اتصالك بالإنترنت (ISP، نقطة WiFi)
  2. الخصم يراقب خادم الوجهة (موقع ويب، خدمة)
  3. يحلل توقيت إرسال واستقبال الحزم
  4. إذا تطابقت الأنماط الزمنية ← يعرف أنك أنت من يتصل بالموقع

🛡️ كيف تحمي نفسك؟

  • ✅ تجنب الأنشطة في أوقات متطابقة بين هويتك الحقيقية والمجهولة

🔴 متقدم
تسريب البيانات بأنواعه الثلاثة (Data Leaks)

من توثيق Qubes الرسمي: هناك 3 أنواع من تسريب البيانات:

  • النوع 1 (Intentional): برمجيات خبيثة تنشئ قنوات خفية تعاونية. ❌ لا حل عملي كامل.
  • النوع 2 (Intentional Sniffing): تخمين مواد المفاتيح عبر القنوات الجانبية. ✅ الحل: إيقاف VMs الأخرى.
  • النوع 3 (Unintentional): برمجيات غير خبيثة تسرب البيانات. ✅ الحل: جدار الحماية + NetVM فارغ.
تصحيح من توثيق Qubes: إيقاف VMs الأخرى لا يحمي من النوع 1. البرمجيات الخبيثة تخزن البيانات وتنتظر عودة VMs المتصلة بالشبكة.
الحل: النوع 1 يمثل حدود نموذج العزل في Qubes.
🔴 متقدم
القنوات الخفية التعاونية (Cooperative Covert Channels)

أخطر تهديد على Qubes. برمجيات خبيثة في VM معزول وأخرى في VM متصل بالشبكة تتواصل عبر CPU cache. جدار الحماية لا يمنع هذا. توثيق Qubes يؤكد عدم وجود حلول عملية كاملة لهذا الهجوم.

الحل: لا يوجد حل تقني كامل. هذه حدود نموذج العزل.
🛡️ حلول Qubes المتقدمة: Ephemeral VM + RAM-VM + Live Mode

تنبيه: هذه هي الدفاعات العملية الوحيدة المتاحة حالياً في Qubes OS ضد التهديدات التي تستهدف القرص الصلب (التحليل الجنائي) والنظام المضيف (dom0). انتبه للفرق بين Ephemeral VM و RAM-VM - فالأولى تشفير زائل على القرص (يُورّث من قالب DVM)، والثانية VM بالكامل في RAM عبر tmpfs.

🧠 Ephemeral VM (وراثة من DVM)

الآلية: تعيين root rw False على قالب DVM ← الأجهزة المشتقة ترث تلقائياً خاصية التشفير الزائل على القرص.

أمر التفعيل:

qvm-volume config DVM_TEMPLATE:root rw False

يحل:

  • 🗑️ Forensic Analysis: لا يمكن استعادة البيانات بعد الإغلاق
  • 📋 تسريب النوع 3: سجلات المستخدم تختفي
لا يحمي من Cold Boot (البيانات على القرص).
🧠 RAM-VM (tmpfs)

الآلية: VM بالكامل في RAM عبر RAM pool + tmpfs.

أمر التفعيل:

qvm-create -P ram_pool -t TEMPLATE VMNAME

🔑 ملاحظة: جميع الأجهزة المستهلكة (Disposable VMs) المشتقة من VM في RAM pool تعمل أيضاً في RAM تلقائياً.

يحل:

  • 🗑️ Forensic Analysis: لا أثر على القرص
  • 💾 تحليل SSD: لا يتفاعل مع القرص
  • 🧬 DVMs المشتقة: تعمل في RAM تلقائياً
لا يحمي من Cold Boot: الهجوم يستهدف RAM نفسها. للحماية من Cold Boot، استخدم BusKill (luksSuspend) أو Shutdown كامل + RAM-Wipe.
⚡ Live Mode (OverlayFS)

الآلية: dom0 يُركب للقراءة فقط. كل التغييرات على dom0 تُكتب في RAM وتختفي عند الإغلاق.

أمر التفعيل: إضافة rootovl إلى GRUB_CMDLINE_LINUX في /etc/default/grub.

يحل:

  • 🚪 Evil Maid: لا برمجيات خبيثة دائمة في dom0
  • 🧠 اختراق dom0: النظام يعود نظيفاً كل مرة
  • 🤖 Intel ME/AMD PSP: يحد من قدرتها على زرع rootkits
لا يحمي من: خصم يمتلك root على dom0 المخترق ويصل للقرص الأصلي.
💡 الخلاصة: Ephemeral VM (عبر توريث root rw False من قالب DVM) يحمي من التحليل الجنائي للقرص (Forensic Analysis) - الأسرع للجلسات اليومية. RAM-VM (عبر RAM pool) يوفر حماية قصوى للبيانات - لا أثر على القرص، وجميع الأجهزة المستهلكة المشتقة منه تعمل في RAM تلقائياً. للحماية من Cold Boot: استخدم BusKill (luksSuspend) أو Shutdown كامل + RAM-Wipe - فالهجوم يستهدف RAM نفسها. Live Mode (عبر rootovl في GRUB) يحمي النظام المضيف نفسه من العبث وهجمات الاستمرارية. معاً، يشكلان درعاً كاملاً ضد هجمات الاستمرارية (Persistence Attacks) والكشف الجنائي (Forensics). لكنهما لا يحميان من القنوات الخفية التعاونية (Cooperative Covert Channels) التي تمثل حدود العزل في Qubes.
"Qubes تطمح للحفاظ على حالة حيث المهاجم الذي يتحكم بالشبكة، ويملك root في أحد VMs، ويستطيع استغلال بطاقة الشبكة، ويستطيع استنساخ القرص - لا يزال غير قادر على السيطرة على dom0 أو رؤية البيانات في VMs الأخرى غير المخترقة."
— Jean-Philippe Ouellet، مطور Qubes سابق
"لا تثق بأحد. مزود الخدمة، مزود VPN، شبكة Tor، نظام التشغيل، وحتى جهاز الكمبيوتر الخاص بك. ثق ولكن تحقق."

تنظيف يدوي (تفريغ RAM pool)

لحذف جميع الأجهزة وتفريغ RAM pool يدوياً:

#!/bin/bash
RAM_POOL_PATH="$HOME/ram_pool"
POOL_NAME="ram_pool"

echo "🧹 تنظيف RAM Pool..."

# حذف الأجهزة
for vm in $(qvm-ls --raw-list 2>/dev/null | grep -E "whonix-work-ram|ram-"); do
    echo "  حذف: $vm"
    qvm-kill "$vm" 2>/dev/null
    qvm-remove --force "$vm" 2>/dev/null
done

# حذف pool
qvm-pool remove "$POOL_NAME" 2>/dev/null

# إلغاء التثبيت
mount | grep -q "$RAM_POOL_PATH" && sudo umount "$RAM_POOL_PATH" 2>/dev/null

# حذف المجلد
[ -d "$RAM_POOL_PATH" ] && sudo rm -rf "$RAM_POOL_PATH"

echo "✅ تم التنظيف"
free -h

🔴 فيزيائي
هجمات BadUSB

لا تثق بأي USB غير معروف. يمكن برمجة أي USB ليصبح لوحة مفاتيح وتنفيذ أوامر بصلاحيات root.

الحماية الأساسية: Qubes يعزل USB تلقائياً في sys-usb (VM منفصل عن dom0). هذا يمنع أي جهاز USB ضار من الوصول المباشر لـ dom0.

للإقلاع من USB: إذا كنت تقلع من هارد خارجي، ثبّت usbguard في dom0:

sudo qubes-dom0-update usbguard sudo systemctl enable usbguard sudo usbguard generate-policy > /etc/usbguard/rules.conf

تحذير "USB in dom0 is not restricted" طبيعي عند الإقلاع من USB.

🟡 شبكة
VPN + Tor معاً

التهديد: مزود الخدمة (ISP) يستطيع معرفة أنك تستخدم Tor، حتى لو كان لا يستطيع رؤية محتوى اتصالك.

الحل (مدفوع): استخدم VPN موثوق قبل Tor. المسار: sys-net → sys-vpn → sys-whonix → anon-whonix. أنشئ كيوب VPN واجعله netvm لـ sys-whonix.

البديل (مجاني): استخدم Tor + Bridges بدلاً من VPN. Bridges تخفي حقيقة استخدام Tor عن ISP بدون تكلفة. راجع تبويب 2 (Whonix) لقسم الجسور.

Tor فقط كافٍ لمعظم المستخدمين. VPN → Tor طبقة إضافية لمن يحتاج إخفاء استخدام Tor نفسه.

التهديد: إهمال التحديثات يترك ثغرات معروفة مفتوحة للاستغلال. طبّق التحديثات فور ظهورها.

التحديث الأساسي:

  • dom0: sudo qubes-dom0-update
  • القوالب: sudo apt update && sudo apt upgrade (داخل كل قالب)

التحديث عبر Tor: Qubes يستخدم UpdatesProxy الذي يوجّه تحديثات القوالب عبر sys-whonix تلقائياً.

اختياري: sudo apt install debsums && sudo debsums -c للتحقق من سلامة الحزم.

🔴 إقلاع
لا تستخدم التشغيل المزدوج (Dual Boot)

التهديد: أي نظام آخر (مثل Windows) يمكنه تعديل /boot غير المشفر وتثبيت Bootkit أو استبدال النواة بملف ضار.

المصادر الرسمية: وثائق Qubes و Kicksecure تحذران من التشغيل المزدوج لأن /boot غير مشفر ويمكن التلاعب به.

من منتدى Qubes: "Bootkit يمكنه تثبيت نفسه دون أن يفعل المستخدم أي شيء خاطئ."

الحلول الآمنة: هارد خارجي منفصل لـ Qubes (تفصله مادياً)، جهاز كمبيوتر مخصص، أو نظام وحيد.
🔴 فيزيائي
خطر USB على dom0

التهديد: حتى لو "مررت" USB إلى VM آخر، dom0 يضطر لتشغيل سوائق USB وقراءة Vendor/Product IDs وجدول الأقسام.

ماذا يمكن أن يفعل USB ضار؟

  • انتحال لوحة مفاتيح (Rubber Ducky) وتنفيذ أوامر بصلاحيات root
  • استغلال ثغرات في سوائق USB (buffer overflows)
  • هجمات DMA عبر Thunderbolt/USB-C

وضعك (الإقلاع من USB): الهارد الخارجي موثوق، لكن أي USB آخر يشكل خطراً. لا يمكن تحقيق عزل USB كامل مع الإقلاع من USB.

لا توصل أي USB غير موثوق. إن أمكن، استخدم وحدتي تحكم USB منفصلتين: lspci | grep USB.
🔴 فيزيائي
هجمات Evil Maid و Cold Boot

① Evil Maid (الخادمة الشريرة):

شخص يصل لجهازك في غيابك (فندق، مكتب، حدود) ويقوم بـ: تثبيت Keylogger، استنساخ القرص، تعديل BIOS أو Bootloader، تثبيت برمجيات على مستوى Firmware.

الحماية: كلمة مرور BIOS + Secure Boot + Anti Evil Maid (AEM - Intel CPUs) + Live Mode (OverlayFS) - dom0 في RAM، النظام يعود نظيفاً كل مرة، لا يمكن زرع برمجيات دائمة + طلاء أظافر على البراغي.

② Cold Boot (الإقلاع البارد):

المهاجم يبرد RAM (هواء مضغوط، نيتروجين سائل) لاستخراج مفاتيح التشفير بعد إطفاء الجهاز بدقائق. هذا الهجوم غير عملي خارج المختبر.

ما يحمي:

  • ✅ BusKill / Dead Man's Switch: luksSuspend يمسح مفاتيح LUKS من RAM + luksErase يدمر رأس التشفير - ثم يغلق النظام.
  • ✅ Shutdown كامل + RAM-Wipe: Dracut RAM-Wipe يمسح الذاكرة عند الإطفاء العادي.
  • ✅ DDR5 RAM: أسرع في فقدان البيانات بعد انقطاع التيار.

ما لا يحمي:

  • ❌ Ephemeral VM: مفتاح التشفير في RAM أثناء الجلسة.
  • ❌ RAM-VM: كل بيانات الجلسة في RAM - هدف مباشر.

🔥 طوارئ
خطة الطوارئ

① إذا احترقت (هويتك انكشفت):

لديك وقت: احذف الهوية من الإنترنت، تخلص من SIM والهاتف (دمره وألقه)، امسح القرص أو دمره فيزيائياً، احتفظ برقم محامٍ، تظاهر بأن شيئاً لم يكن.

لا وقت (طارئ): أغلق/افصل التيار فوراً. RAM ستتلاشى والبيانات تصبح غير قابلة للقراءة. التزم الصمت - لا تتحدث بدون محامٍ.

② إذا اكتشفت تلاعباً:

باللابتوب: لا تلمس الزرع - لا تستخدم الجهاز. أخرج القرص (إذا كان مشفراً) وغادر فوراً.

بالغرفة: افترض أن كل شيء مزروع (كاميرات، مسجلات). لا تلمس شيئاً - غادر فوراً بهدوء. لا تعد أبداً.

③ حيلة لكشف التفتيش: التقط صورة لمكتبك قبل المغادرة وقارن عند العودة. خيط رفيع أو قطعة شمع على الدرج.

④ استعادة البيانات: qubes-backup --restore على جهاز جديد نظيف.

اعرف حقوقك: خريطة قوانين التشفير العالمية. إذا كان خصمك قادراً على الوصول الفيزيائي، فالخيار الوحيد هو الاختفاء الفوري.
💡 نصائح
نصائح أمنية سريعة

① أساسيات: كلمات مرور قوية وفريدة لكل خدمة. حدّث النظام أسبوعياً. استخدم Disposable VMs لفتح الملفات المشبوهة.

② شبكة وتصفح: عزل USB في sys-usb (افتراضي في Qubes). تعطيل JavaScript أو استخدام NoScript. استخدم متصفح Tor داخل Whonix للأنشطة الحساسة.

③ تخزين وتشفير: LUKS2 مع integrity للبيانات الحساسة. RAM pool للتشغيل المؤقت (لا أثر ع القرص). معلمات النواة: init_on_free=1, init_on_alloc=1.

تجنب تحميل البرامج من مصادر غير موثوقة. استخدم المتاجر الرسمية والمستودعات الموقعة فقط.
🛡️ فيزيائي
حماية فيزيائية إضافية

① حماية الإقلاع: كلمة مرور BIOS/UEFI قوية - تمنع تغيير إعدادات الإقلاع. فعّل Secure Boot. استخدم TPM لتخزين مفاتيح التشفير بأمان.

② أدوات متقدمة: Anti Evil Maid (AEM) - متوفر في Qubes (Intel CPUs). Heads firmware - بديل BIOS مفتوح مع TPM verification كامل - أقوى من Secure Boot.

③ أجهزة متخصصة: NitroPad (Nitrokey) - لابتوب مع Heads مدمج. Purism Librem - لابتوب مع kill switches فيزيائية.

④ أثناء التنقل: لا تترك جهازك دون مراقبة. استخدم Faraday Cage لمنع الاتصالات اللاسلكية. تحقق من الأجهزة الغريبة عند العودة.

إذا كان خصمك حكومة أو وكالة استخبارات، حتى هذه الإجراءات قد لا تكون كافية.

🧬 Kicksecure - توحيد البصمة وتحصين النظام

ما هو Kicksecure؟

Kicksecure هو نظام تشغيل مبني على Debian، معدّل ليكون شديد التحصين (Hardened) ضد هجمات التعرف على الجهاز. هدفه الرئيسي هو توحيد المعلومات (Normalization) التي يمكن للتطبيقات (خاصة المتصفحات) قراءتها.

النتيجة: جميع مستخدمي Kicksecure يبدون متطابقين إحصائياً، مما يجعل تتبع الفرد عبر بصمة المتصفح مستحيلاً.

Kicksecure داخل Qubes OS

في Qubes OS، يمكن استخدام Kicksecure كـ Template (قالب) لإنشاء AppVMs. بهذه الطريقة:

  • كل AppVM تحصل على حماية Kicksecure تلقائياً
  • يمكن توجيهها عبر sys-whonix للحصول على Whonix + Kicksecure معاً
  • يتم العزل الكامل بين الهويات المختلفة
أفضل ممارسة: qvm-create --template=kicksecure anon-work && qvm-prefs anon-work netvm sys-whonix
لماذا Kicksecure حتى مع Tor؟

Tor يخفي عنوان IP فقط. لكن المواقع يمكنها التعرف عليك عبر بصمة المتصفح: الخطوط، دقة الشاشة، الإعدادات... Kicksecure يوحد هذه المعلومات بين جميع مستخدميه، مما يجعل بصمتك غير مميزة.

📥 التثبيت والإعداد الأساسي

تثبيت قالب Kicksecure

الخطوة 1: تثبيت القالب من مستودعات المجتمع:

qvm-template --enablerepo qubes-templates-community install kicksecure-18

الخطوة 2: تثبيت الإضافة الرسمية لإدارة الوقت:

sudo qubes-dom0-update --action=install qubes-core-admin-addon-kicksecure

الخطوة 3: تحديث القالب دورياً:

sudo qubes-dom0-update --action=update kicksecure
حول الاستقرار: القالب يُصنف "قيد التطوير" لكن هذا تحذير احترازي فقط. المشاكل نادرة جداً (أقل من 1-2%) وتتمحور غالباً حول إشعارات بسيطة قابلة للحل. الاستخدام اليومي مستقر تماماً.
🛡️ tirdad - حماية إضافية لاتصالات TCP

ما هي tirdad؟ أداة تقوم بعشوائية أرقام تسلسل TCP (TCP ISN Randomization)، مما يصعّب على المهاجمين حقن حزم ضارة في اتصالاتك.

  3. أغلق القالب وأعد تشغيل AppVMs المبنية عليه.

المشكلة: تطبيقات مثل Cwtch و OnionShare قد تحتوي على Tor مدمج. في Whonix، هذا يسبب مشكلة "تكديس Tor" (Tor-over-Tor) وقد يتم حظر الاتصال. الحل الصحيح يتطلب إعدادات في كل من البوابة ومحطة العمل، مع الانتباه لمشكلة الملفات المضغوطة (.gz) التي قد تعطل الخدمة.

torsocks -i ./application.AppImage

• الإعدادات ← Network & Servers
• ☑ Use SOCKS proxy (127.0.0.1:9050)
• ☑ Use .onion hosts (يظهر بعد تفعيل SOCKS)
🧅 خيارات Use .onion hosts:
• No: لا يستخدم عناوين .onion أبداً
• When Available (موصى به): يستخدم .onion إذا كان متاحاً، وإلا يستخدم العناوين العادية
• Required (خصوصية قصوى): يفرض استخدام .onion فقط، ولن يتصل بالخوادم العادية
⚠️ تنبيهات هامة:
  • خيار .onion hosts مخفي بدون SOCKS: حالياً، خيار Use .onion hosts لا يظهر في واجهة SimpleX إلا بعد تفعيل Use SOCKS proxy. هناك طلب مجتمعي لإتاحة هذا الخيار بشكل مستقل لمستخدمي Whonix، لأن Whonix يمرر كل الاتصالات عبر Tor على مستوى النظام.
  • حسب تصريح فريق SimpleX: بشكل افتراضي، يمكن لخوادم المراسلة ملاحظة عناوين IP الخاصة بالمستخدمين. تُستخدم عناوين .onion فقط عندما يستخدم المستخدمون Tor مع تفعيل SOCKS proxy.
  • TransPort بدون SOCKS: جميع التطبيقات غير المضبوطة تتشارك نفس دائرة Tor، مما قد يؤدي إلى ارتباط الهويات الرقمية.
  • torsocks -i: يعمل بشكل موثوق على Qubes OS (الذي يستخدم X11 عبر طبقة GUI الافتراضية). قد يسبب أعطالاً على أنظمة Wayland مثل Whonix المستقل بسبب تداخله مع استدعاءات النظام المحلية.

Cwtch (⭐ الموصى به - ند للند مقاوم للبيانات الوصفية)

Cwtch (يُنطق /kʊtʃ/) هو بروتوكول مراسلة لامركزي ومقاوم للبيانات الوصفية (Metadata)، مبني على شبكة Tor v3 Onion Services. تم تطويره بواسطة منظمة Open Privacy Research Society (كندا).

المميزات:
  • 🔒 ند للند (P2P حقيقي) - اتصال مباشر دون خوادم وسيطة
  • 🧅 Tor مدمج - إخفاء تام لعنوان IP
  • 🎭 هويات متعددة (Profiles) - شخصيات معزولة تماماً
  • 📦 محمول بالكامل - يعمل من مجلد واحد (USB)
  • 🔐 Perfect Forward Secrecy (PFS)
  • 🛡️ مقاوم للبيانات الوصفية
  • 🚫 الإنكار الآمن (Plausible Deniability): يتيح بروتوكول المصادقة في Cwtch إنكار حدوث المحادثة تماماً بعد إغلاقها. يمكن لأي طرف تزوير سجلات المحادثة بالكامل، مما يجعل من المستحيل إثبات أن المحادثة "الحقيقية" قد حدثت. هذا يجعله الخيار الوحيد الذي يوفر هذه الحماية بين تطبيقات المراسلة.
القيود:
  • 🐢 يتطلب اتصالاً متزامناً (الطرفان معاً) للمحادثات الخاصة
  • 📱 غير متوفر على iOS (Android و Desktop فقط)
  • 🔧 يتطلب إعدادات خاصة على Whonix/Qubes
  • 📊 قاعدة مستخدمين صغيرة: ~5,000+ تحميل على Android، وغير معروف على Desktop (تحميل مباشر)
  • 🔍 لم يخضع لتدقيق أمني مستقل بعد (حسب صفحة Whonix الرسمية)

# 1ب. إضافة معلمات النواة لجميع الـ VMs فقط

setup_kernel_opts_vms() {
    clear
    echo "=========================================="
    echo "  🔧 إضافة معلمات النواة لجميع الـ VMs"
    echo "=========================================="

    # VMs لا ترث معلمات GRUB من dom0، لذا يجب تعيينها بشكل منفصل
    NEW_OPTS="init_on_free=1 init_on_alloc=1 mitigations=auto random.trust_cpu=off random.trust_bootloader=off spec_store_bypass_disable=on tsx_async_abort=full mds=full,nosmt page_alloc.shuffle=1 randomize_kstack_offset=on nosmt=force l1tf=full,force"

    echo "📋 سيتم إضافة هذه المعلمات لجميع VMs:"
    echo "   $NEW_OPTS"
    echo ""
    echo "🎭 سيتم تفعيل anon-timezone"
    echo ""
    echo "⏭️  dom0: يتم التخطي (له معلماته الخاصة في GRUB)"
    echo "⏭️  sys-whonix: يتم التخطي (بوابة Whonix لا تحتاج هذه المعلمات)"
    echo ""

    read -p "متابعة؟ (y/n): " confirm
    [[ "$confirm" != "y" && "$confirm" != "Y" ]] && { err "تم الإلغاء"; pause; return 0; }

    info "إغلاق جميع VMs قبل تطبيق المعلمات..."
    for v in $(qvm-ls --raw-list 2>/dev/null | grep -v dom0); do
        echo "   🔻 إغلاق: $v"
        qvm-shutdown --wait "$v" 2>/dev/null || true
    done
    ok "تم إغلاق جميع VMs"

    for vm in $(qvm-ls --raw-list 2>/dev/null); do
        vm_clean=$(echo "$vm" | tr -d '[:space:]')
        
        if [ "$vm_clean" = "dom0" ]; then
            echo "   ⏭️  $vm_clean: تم التخطي (يطبق عبر GRUB)"
            continue
        fi
        
        if [[ "$vm_clean" =~ ^sys-whonix ]]; then
            echo "   ⏭️  $vm_clean: تم التخطي (بوابة Whonix)"
            continue
        fi
        
        CURRENT=$(qvm-prefs "$vm_clean" kernelopts 2>/dev/null | tr '\n' ' ' | sed 's/^[[:space:]]*//;s/[[:space:]]*$//')

        if [ -z "$CURRENT" ] || [ "$CURRENT" = "default" ]; then
            qvm-prefs "$vm_clean" kernelopts "$NEW_OPTS"
            echo "   ✅ $vm_clean: تم تعيين معلمات جديدة"
        else
            FINAL="$CURRENT"
            for opt in $NEW_OPTS; do
                if ! echo " $FINAL " | grep -q " $opt "; then
                    FINAL="$FINAL $opt"
                fi
            done
            qvm-prefs "$vm_clean" kernelopts "$FINAL"
            echo "   ✅ $vm_clean: تم الدمج"
        fi
        
        qvm-features "$vm_clean" anon-timezone 1 2>/dev/null
        
    done

    echo ""
    info "إعادة تشغيل sys-net و sys-usb..."
    for vm in sys-net sys-usb; do
        if qvm-check --quiet "$vm" 2>/dev/null; then
            qvm-start "$vm"
            echo "   🔄 $vm"
        fi
    done

    warn "⚠️  تذكر: أي VM كان مفتوحاً أثناء تشغيل السكربت يجب إعادة تشغيله يدوياً لتطبيق المعاملات الجديدة."

    ok "تم تطبيق جميع الإعدادات على الـ VMs"
    pause
}

# 2. MAC Randomization + تعطيل IPv6 + إغلاق القالب

mac_random() {
    info "إعداد MAC Randomization: $BASE_TEMPLATE"
    
    info "التحقق من إعدادات IPv6 لـ sys-net و sys-firewall..."
    
    for vm in sys-net sys-firewall; do
        if qvm-ls "$vm" >/dev/null 2>&1; then
            CURRENT_OPTS=$(qvm-prefs "$vm" kernelopts 2>/dev/null | xargs)
            
            if [[ "$CURRENT_OPTS" == *"ipv6.disable=1"* ]]; then
                warn "⚠️ $vm: IPv6 معطل بالفعل"
            else
                info "🔧 $vm: جاري تعطيل IPv6..."
                qvm-prefs "$vm" kernelopts "$CURRENT_OPTS ipv6.disable=1"
                ok "✅ تم تعطيل IPv6 لـ $vm"
            fi
        else
            warn "$vm غير موجود، تخطي"
        fi
    done
    
    if qvm-ls "$BASE_TEMPLATE" >/dev/null 2>&1; then
        info "🔧 إعداد MAC Randomization في $BASE_TEMPLATE..."
        qvm-run "$BASE_TEMPLATE" "sudo bash -c 'mkdir -p /usr/lib/NetworkManager/conf.d/ && cat > /usr/lib/NetworkManager/conf.d/31-randomize-mac.conf << EOF
[connection]
ethernet.cloned-mac-address=random
wifi.cloned-mac-address=random
wifi.scan-rand-mac-address=yes
EOF'"
        ok "✅ تم إعداد MAC Randomization في $BASE_TEMPLATE"
        
        info "إغلاق القالب $BASE_TEMPLATE..."
        qvm-shutdown "$BASE_TEMPLATE" 2>/dev/null || true
        ok "تم إغلاق $BASE_TEMPLATE"
    else
        warn "القالب $BASE_TEMPLATE غير موجود، تخطي إعداد MAC"
    fi
    
    info "ملاحظة: قد تحتاج إلى إعادة تشغيل الأجهزة المبنية على هذا القالب"
    pause
}

    # ============================================
